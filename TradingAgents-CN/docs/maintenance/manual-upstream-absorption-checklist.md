# 人工上游吸收清单

本文档用于记录 TradingAgents-CN 在与上游项目存在较大分叉时，哪些模块已经完成吸收，哪些属于兼容保留层，哪些暂时不再继续追求与上游完全一致。

## 1. 当前原则

- 不再做整仓自动同步
- 采用按模块、按风险、按收益的人工作业方式吸收上游更新
- 优先同步核心能力、底层抽象和明确的缺陷修复
- 中文文档体系、数据库配置体系、Web 管理能力保持本地演化

## 2. 已完成吸收

### LLM 相关

- `llm_clients` 抽象层已接入主链路
- 共享模型目录已接入 CLI 与轻量校验
- provider 规范键已统一为 canonical key：
  - `qwen`
  - `glm`
  - `openai`
  - `google`
  - `deepseek`
  - `openrouter`
  - `ollama`
  - `qianfan`
  - `custom_openai`
- `trading_graph.py` 的主要 provider 初始化路径已大幅收口到 `llm_clients`
- `fundamentals_analyst.py` 中与 qwen 相关的 fresh llm 重建逻辑已适配新路径

### 配置与模型目录

- provider 默认 URL / 环境变量映射已统一
- provider 别名归一化工具已建立
- model catalog 校验已统一到共享目录

### 数据库与运维

- MongoDB 默认库名已调整为 `tradingagentscn`
- 数据库支持按大版本 / 实例隔离命名
- 开发环境共享库保护已加入
- 数据库迁移脚本、provider 规范化脚本、运维文档已补齐

## 3. 兼容保留层

以下内容仍然保留，但定位已从“主链路实现”转为“兼容层/特化层”：

- `tradingagents/llm_adapters/`
  - 其中 Google 兼容适配器仍作为底层实现存在
  - 业务层已尽量通过 `llm_clients` 间接使用它，而不是直接依赖
- `GoogleToolCallHandler`
  - 当前仍有明确业务价值，用于处理 Google 模型工具调用行为差异

## 4. 暂不继续追求完全对齐

以下部分当前不以“完全贴上游”为目标：

- 中文文档体系
- Web 后端配置管理体系
- MongoDB / Redis / 多环境隔离策略
- 中国市场、本地化数据源和中文增强功能
- 运营、部署、便携版相关脚本与文档

这些能力已经形成明显本地分叉，继续强行对齐上游收益较低、风险较高。

## 5. 后续可继续吸收的方向

如果未来继续人工吸收上游，建议按下面顺序评估：

1. `tradingagents/` 下核心框架层的小型缺陷修复
2. 与 LLM 抽象、模型目录、工具调用稳定性相关的改进
3. 与数据流、缓存、性能相关，但不破坏本地配置体系的优化
4. 文档和示例中具有直接教学价值的内容

## 6. 暂停线

出现以下情况时，不建议继续大规模吸收上游：

- 上游变更需要重写当前数据库配置体系
- 上游实现直接破坏中文增强功能或多市场支持
- 上游变更与当前 Web 架构冲突明显
- 收益主要是“代码看起来更像上游”，而不是功能/稳定性提升

## 8. 使用方式

后续每次准备吸收上游更新时，先回答三个问题：

1. 这次改动属于“核心能力”还是“本地分叉区”？
2. 能否拆成独立的小批次提交？
3. 是否能在现有回归测试范围内验证？

如果三个问题里有两个答案是否定的，建议先不要吸收。
